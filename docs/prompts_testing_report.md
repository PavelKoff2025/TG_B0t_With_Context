# Отчёт: тестирование трёх промптов из урока

## Цель

Протестировать промпты из папки `prompts/` и сравнить ответы при **двух значениях temperature** (0.2 и 0.8). Подготовить выводы по каждому промпту.

## Методика

| Параметр | Значение |
|----------|----------|
| Инструмент | `run_temperature_tests.py` / OpenAI API |
| Модель | `gpt-4o-mini-2024-07-18` |
| Temperature A | **0.2** (стабильность) |
| Temperature B | **0.8** (вариативность) |
| max_tokens | `2000` |
| Вход | `test_input` из каждого JSON |
| Дата | 17.07.2026 |

Критерии: формат Markdown, полнота ответа, разница между temperature.

Артефакты: `docs/prompt_tests/` (`*_temp_0_2.md`, `*_temp_0_8.md`).

---

## 1. Промпт «Резюме текста» (`01_summary_prompt.json`)

### Результаты

| Temperature | Markdown | JSON | Длина (симв.) | Токены (вход/выход/всего) |
|-------------|----------|------|---------------|---------------------------|
| **0.2** | да | нет | 982 | 951 / 241 / **1192** |
| **0.8** | да | нет | 849 | 951 / 216 / **1167** |

### Сравнение ответов

Оба ответа сохраняют структуру преподавателя:
`# заголовок` → `## Основная идея` → `## Ключевые пункты` → `## Важные детали` → `## Выводы`.

| Аспект | temp 0.2 | temp 0.8 |
|--------|----------|----------|
| Структура | Полная, 5 секций | Полная, 5 секций |
| Ключевые пункты | 5 пунктов, более детально | 4 пункта, чуть короче |
| Стабильность | Более «учебный», сухой стиль | Формулировки чуть свободнее |
| Смысл | Все ключевые идеи исходника сохранены | То же |

### Выводы по промпту 1

1. Промпт **стабильно** держит Markdown-формат при обеих temperature.
2. При **0.2** резюме чуть полнее и ровнее по пунктам.
3. При **0.8** ответ короче, формулировки разнообразнее, но смысл тот же.
4. **Рекомендация:** для резюме лучше **temperature 0.2–0.3** (предсказуемость важнее креатива).

## 2. Промпт «Генерация структуры кода» (`02_code_structure_prompt.json`)

### Результаты

| Temperature | Markdown | Дерево `├──` | Длина (симв.) | Токены (вход/выход/всего) |
|-------------|----------|--------------|---------------|---------------------------|
| **0.2** | да | да | 3878 | 999 / 1171 / **2170** |
| **0.8** | да | да | 3207 | 999 / 941 / **1940** |

### Сравнение ответов

Оба варианта: абзац про FastAPI + React + PostgreSQL + JWT, затем `### Структура директорий` и дерево.

| Аспект | temp 0.2 | temp 0.8 |
|--------|----------|----------|
| Полнота дерева | Богаче: notifications API, больше models/services | Короче, часть веток свёрнута (`...`) |
| Комментарии `#` | Подробные | Есть, но меньше деталей |
| Полезность | Готовый скелет проекта «под ключ» | Достаточно для обзора, но менее детально |

### Выводы по промпту 2

1. Формат как у преподавателя соблюдён при обеих temperature.
2. **0.2** даёт более полное и полезное дерево (лучше для старта разработки).
3. **0.8** короче и чуть «ленивее» в деталях.
4. **Рекомендация:** для структуры кода — **temperature 0.2**.

## 3. Промпт «Планирование задач» (`03_task_planning_prompt.json`)

### Результаты

| Temperature | Markdown | Длина (симв.) | Токены (вход/выход/всего) |
|-------------|----------|---------------|---------------------------|
| **0.2** | да | 2753 | 1204 / 785 / **1989** |
| **0.8** | да | 2919 | 1204 / 785 / **1989** |

### Сравнение ответов

Оба плана на 8 недель с секциями: цель, этапы, декомпозиция, зависимости, приоритеты, оценка, ресурсы, риски, чекпоинты.

| Аспект | temp 0.2 | temp 0.8 |
|--------|----------|----------|
| Этапы | Исследование → auth → проекты/задачи → тест | Проектирование → auth → проекты/задачи → тест |
| Декомпозиция | Больше задач в этапе 3 (включая уведомления) | Чуть укрупнённые задачи |
| Приоритеты | Must/Should/Nice — корректно | Must/Should/Nice — корректно |
| Разница | Более детальный backlog | Ближе по длине, но названия этапов другие |

### Выводы по промпту 3

1. Markdown-план стабилен при обеих temperature.
2. Разница в основном в **формулировках этапов** и степени детализации задач.
3. Токены совпали (1989), длина чуть больше при 0.8.
4. **Рекомендация:** для планирования лучше **0.2** (меньше «плавания» в структуре этапов); **0.8** полезен, если нужны альтернативные варианты плана.

---

## Общие выводы

1. **Все три промпта работают** и при 0.2, и при 0.8 выдают нужный Markdown (не JSON).
2. **Низкая temperature (0.2)** в среднем даёт более полные, стабильные и «сдаваемые» ответы.
3. **Высокая temperature (0.8)** добавляет вариативность формулировок, иногда сокращает детализацию (особенно у структуры кода).
4. Практическая рекомендация для урока:
   - резюме / структура кода / план задач → **0.2–0.3**;
   - brainstorm / альтернативы → **0.7–0.8**.
5. Явный `format.template` в JSON критичен: без него модель склонна уходить в JSON.
